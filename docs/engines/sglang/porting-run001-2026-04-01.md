# SGLang ROCm Porting Run 001

**Date:** 2026-04-01  
**Target:** `THOTH/forks/sglang`  
**Hardware mode:** RX 6700 XT via `HSA_OVERRIDE_GFX_VERSION=10.3.0`

## Scope

Bring SGLang up as the real ROCm target for:

- TurboQuant KV cache
- EAGLE / EAGLE3 runtime validation
- PrismML Bonsai 1-bit GGUF support
- OpenCoder target-model bring-up

## Source Changes Landed

### TurboQuant / ROCm

- Cherry-picked the AMD TurboQuant KV patch set from upstream PR `#21628`
- Patched `sgl-kernel/setup_rocm.py` so ROCm builds target `gfx1030` and normalize `gfx1031` / `gfx1035`
- Built `sgl-kernel` successfully inside Docker and confirmed `sgl_kernel` imports on ROCm

### GGUF / 1-bit

- Added Prism GGUF compatibility shims in `python/sglang/srt/utils/gguf_compat.py`
- Patched the GGUF quantization layer for HIP fallback dequant/matmul behavior
- Patched model-loader integration so Bonsai GGUF files no longer die on unsupported Prism quant enums
- Relaxed ROCm GGUF gating in `model_config.py`

### Runtime robustness

- Added optional import fencing for non-essential quant backends that were breaking ROCm startup
- Added a fail-fast EAGLE3 validation in `server_args.py`:
  - plain instruct checkpoints with no `eagle_config` are rejected immediately
  - real EAGLE3 drafts with `eagle_config` still pass

## Container Validation

### OpenCoder EAGLE3

Attempted configuration:

- target: `infly/OpenCoder-8B-Instruct`
- draft: `infly/OpenCoder-1.5B-Instruct`
- mode: `EAGLE3`
- KV cache: `tq4`

Observed sequence:

1. Initial run failed on a corrupt local 8B shard:
   - `model-00002-of-00004.safetensors`
2. The corrupt shard was backed up and re-fetched from Git LFS
3. After adding the EAGLE3 draft guard, the same runtime now fails immediately with the correct message:

> `EAGLE3 requires a trained EAGLE3 draft checkpoint ... Use STANDALONE for plain draft-model speculation, or provide an EAGLE3 draft artifact.`

This is the right outcome. The OpenCoder `1.5B` instruct checkpoint is not a true EAGLE3 draft artifact.

### Bonsai 1-bit SGLang

Observed sequence:

1. Bonsai `1.7B` GGUF reaches weight loading successfully
2. Weight load completes as `Qwen3ForCausalLM`
3. KV cache allocation succeeds
4. First serve attempt failed in CUDA-graph capture because `tvm_ffi` is absent
5. Current rerun uses `--disable-cuda-graph` to bypass that ROCm-irrelevant blocker

## Important Outcome

SGLang is now the correct runtime home for this work, but OpenCoder still needs a **trained EAGLE3 draft checkpoint** before “OpenCoder in EAGLE3” is a valid runtime claim.

What the current port proves:

- ROCm TurboQuant integration is live in SGLang
- Prism 1-bit GGUF compatibility is live in SGLang
- OpenCoder can be validated for TurboQuant and standalone speculation in SGLang
- True OpenCoder EAGLE3 requires the training layer, not a plain 1.5B instruct checkpoint
